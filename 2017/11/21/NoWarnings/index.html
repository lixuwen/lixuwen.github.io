NoWarnings

发表于 2017-11-21 | 分类于 iOS

请问我们正在开发或维护的项目中有多少个Warning?这些都是Warning而不是Error难道真的不需要管么?相信两位数的Warning应该算少的了吧，对于有代码洁癖的和强迫症的开发者来说，想必真是太虐心了，对于IDE都已经发现的Warning我们怎么能视而不见呢?如果真的没问题IDE又怎么会提示Warning呢?

一、消除Warning

1.暴力解决方法

1.1 全部忽略

TARGETS->Build Settings->Inhibit All Warnings 置为YES

1.2 忽略指定类型的Warning

第1步找到Warning提示

第2步点击”Reveal in Log”显示Warning信息

第3步设置Warning Flags
TARGETS->Build Settings->搜索 Other Warning flags，添加 -Wno-deprecated-declarations

注意:[-Wdeprecated-declarations]，括号中的内容就是这种警告的类型，-W 是打开这种类型的警告,如果我们要关闭某种类型的警告, 要将-W 换成 -Wno-

1.3 忽略指定文件的指定类型的Warning

TARGETS->Build Phases->Compile Sources，找到要忽略的文件添加需要忽略的Warning的Compiler Flags

注意:这里的Compiler Flags可以添加多个,以空格分隔即可

1.4 添加预编译指令

想要忽略哪些代码的哪些警告都可以用这个宏来告诉编译器

2.科学解决方法

暴力解决Warning的方法实际上是掩耳盗铃，虽然Warning是不提示了，但是Warning还是真真实实存在的，最好的方法就是一一解决掉Warning，下面就是我们项目中遇到的Warning，有Warning出现原因和解决方法

2.1 Dependency Analysis Warning

原因：
没有明确指定 @property 的属性，通常是在声明一个指针的时候只写了@property (nonatomic)
解决方法：
明确指定 @property 的属性

原因：
重复的文件或者是重复编译的文件
解决方法：
去重

原因：
ProjectName-Info.plist被添加到了Compile Sources中
解决方法：
找到该文件并从Compile Sources中移除

原因:
在xxxx架构平台上不能处理 filename.ext 这个文件
解决方法:
1、如果是无用文件删除即可
2、检查文件类型是否有误导致不能编译

2.2 Semantic Issue(语义问题)

原因:
大作用域处声明的变量与小作用域内的声明的变量重名了
解决方法:
把小作用域内的变量重新命名

原因：
继承了某一个协议但是没有实现该协议的 @require 方法，默认是 @require 的
解决方法：
1、实现该方法
2、如果该协议的这个方法不是必须要实现的就可以修改成 @optional 的

原因：
在.h中声明了一个方法，但是在.m中没有实现
解决方法：
要么在.m中实现该方法，要不把.h中的声明删除

原因：
未找到protocol的定义，通常情况是在.h中通过@protocol xxxx 声明了一个protocol，在.m中import protocol 的头文件
解决方法：
在.h中import protocol 的头文件

原因：
类型冲突
解决方法：
类型对应

原因：
指针类型不兼容
解决方法：
AAAA与BBBB的类型要一样

原因：
属性名称错误，例如，大写搞成小写，或者反之
解决方法：
1、自己手动修改属性名称
2、借助Xcode自动修复

原因：
参数类型不匹配
解决方法：
修改成统一的参数类型

原因：
方法的声明与实现中的返回值类型不匹配
解决方法：
返回值类型要统一

原因：
方法xxxx的声明定义或者实现缺失
解决方法：
如果不需要这个方法就删除其声明或者实现；如果需要这个方法就添加其声明或者实现

原因：
property ‘xxxx’已经在父类中声明了，子类中重复声明
解决方法：
1、子类中用 @dynamic 来标明
2、在子类中不做 @property 声明

原因：
在ARC模式下，运行时需要知道如何处理你正在调用的方法的返回值。这个返回值可以是任意值，如void int char NSString id等等。ARC通过头文件的函数定义来得到这些信息。所以平时我们用到的静态选择器就不会出现这个警告。因为在编译期间，这些信息都已经确定。但是通过 NSSelectorFromString(@"someMethod") 生成SEL时ARC并不知道该方法的返回值是什么。
解决方法：
https://github.com/dabing1022/Blog/issues/2

原因：
本质上是类型不匹配
解决方法：
类型要明确统一

原因：
AAAA类有重复的category定义
解决方法：
1、修改category的名字
2、去掉重复的category类

2.3 Value Conversion Issue

原因：
隐式转换，精度缺失
解决方法：
是什么类型就用什么类型

2.4 Format String Issue

原因：
格式化类型不匹配
解决方法：
修改格式化类型对应一致

原因：
向一个需要非空参数的方法传入一个空的参数;往往是偷懒为了调用一些类的代理/协议方法
解决方法：
别偷懒，把要调用的方法重新封装，多处调用即可

2.5 ARC Retain Cycle

原因及解决方法:
见Apple文档说明

2.6 Deprecations

原因：
方法在某些版本上被弃用
解决方法：
1、用新方法进行替换
2、如果新方法只能在高版本使用并且还需要支持低版本应用，那么就用版本区分来分别使用新方法和旧方法

2.7 Unused Entity Issue

原因：
声明的变量未被使用
解决方法：
1、删除
2、注释掉，添加说明

2.8 Warning limit exceeded

原因：
某一个文件中的warning超过200个上限了
解决方法：
这么多warning大多是同一类问题导致的

2.9 Lexical or Preprocessor Issue

原因：
宏重定义了
解决方法：
1、去重
2、如果自己写的类库中用到了宏定义，最好加上本类的前缀/后缀；如果是工程中通用的宏，可以在添加工程的前缀/后缀

2.10 User-Defined Issue

原因：
开发者自己添加的warning
解决方法：
删除，添加注释说明

2.11 Apple Mach-O Linker Warning

原因：
编译时链接不到对应的文件目录
解决方法：
TARGETS -> Build Settings -> Search Paths -> * Search Paths 中找到对应的目录移除

二、添加Warning

既然我们有能力去掉Warning了，那么有时我们也需要添加Warning

1. 添加 -W#warnings 类型的警告

2. 添加 -W#pragma-messages 类型的警告

时间都去哪了
